# Участие в разработке AIBRO Business

Спасибо, что рассматриваете возможность участия в развитии AIBRO Business! В этом документе описаны все необходимые сведения для эффективного участия.

## Кодекс поведения

Прежде чем начать, пожалуйста, ознакомьтесь с нашим [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md).

## Как внести вклад

### 1. Сообщения об ошибках

#### Перед созданием баг-репорта:
- Проверьте, не сообщалось ли уже об этой проблеме
- Убедитесь, что проблема воспроизводится в последней версии
- Подготовьте минимальный пример для воспроизведения ошибки

#### При создании баг-репорта:
- Опишите воспроизводимый сценарий
- Опишите ожидаемое поведение
- Опишите фактическое поведение
- Предоставьте контекст (браузер, ОС, версии)
- Приложите скриншоты или логи, если применимо

### 2. Предложения улучшений

Мы рады любым предложениям по улучшению функциональности, производительности, безопасности или UX.

#### Для предложений:
- Опишите текущую ситуацию
- Объясните, почему предложение улучшит проект
- Опишите возможные методы реализации
- Рассмотрите возможные побочные эффекты

### 3. Пулл реквесты

#### Перед созданием PR:
- Убедитесь, что вы работаете с последней версией кода
- Создайте отдельную ветку для ваших изменений
- Опишите изменения в commit сообщениях

#### Требования к коду:
- Код должен соответствовать стиль руководства
- Все тесты должны проходить
- Новый функционал должен быть покрыт тестами
- Код должен проходить линтинг

## Настройка окружения

### Требования
- Node.js 18+
- npm или yarn
- Git

### Установка
```bash
# Клонируйте репозиторий
git clone https://github.com/Sskutushev/AIRBRO.git
cd AIRBRO

# Установите зависимости для фронтенда
npm install

# Установите зависимости для бэкенда
cd backend
npm install

# Настройте переменные окружения (см. .env.example файлы)
```

### Запуск в режиме разработки
```bash
# Для запуска backend:
cd backend
npm run dev

# В отдельном терминале запустите frontend:
cd ..
npm run dev
```

## Стиль кода

### TypeScript/JavaScript
- Используйте TypeScript для всей новой функциональности
- Используйте строгую типизацию (strict mode)
- Используйте современные возможности ES
- Используйте функциональные компоненты и хуки в React
- Используйте асинхронные паттерны (async/await)

### React
- Следуйте принципам компонентного подхода
- Используйте memo, useCallback, useMemo когда необходимо
- Используйте хуки для логики
- Используйте композицию компонентов
- Используйте TypeScript для определения пропсов

### Именование
- Используйте camelCase для переменных и функций
- Используйте PascalCase для компонентов
- Используйте UPPER_CASE для констант
- Используйте описательные имена

### Коммиты
Мы используем conventional commits:
```
<type>(<scope>): <subject>

<body>

<footer>
```

Примеры:
- `feat(auth): add login with social providers`
- `fix(api): resolve CORS issue in payment endpoint`
- `docs(readme): update installation instructions`

## Тестирование

### Требования к тестам
- Все нововведения должны быть покрыты юнит тестами
- Все компоненты должны быть покрыты интеграционными тестами
- Тесты должны быть читаемыми и сопровождаемыми
- Тесты должны покрывать как успешные, так и неудачные сценарии

### Типы тестов
- **Unit tests** - для тестирования отдельных функций и компонентов
- **Integration tests** - для тестирования взаимодействия между компонентами
- **API tests** - для тестирования API endpoints
- **E2E tests** - для тестирования пользовательских сценариев

### Запуск тестов
```bash
# Запуск всех тестов
npm run test

# Запуск тестов с покрытием
npm run test:coverage

# Запуск тестов с UI
npm run test:ui
```

## Релизы

Мы используем semantic versioning (MAJOR.MINOR.PATCH):
- MAJOR версия - для несовместимых изменений API
- MINOR версия - для новых функций, обратно совместимых
- PATCH версия - для исправлений ошибок, обратно совместимых
